# Процесс разработки

Любой новый функционал **сначала описывается, согласуется и только потом реализуется**.

## 1) Структура документации

Вся документация живёт в `docs/`. На каждый раздел сервиса — своя папка:

```
docs/
  <Раздел>/
    <Раздел>.md                       # главное ТЗ: как раздел работает СЕЙЧАС
    changes/
      2026-06-06-короткое-название.md  # одно изменение = один файл
      ...
  general/
    changes/
      2026-06-06-короткое-название.md  # изменения без привязки к разделу
      ...
```

- **Папка раздела** = раздел сервиса (например, `Бронирование/`, `Залы/`).
- **Главный файл `<Раздел>.md`** — функциональное ТЗ актуального состояния раздела.
  Его всегда можно открыть и прочитать, как раздел работает прямо сейчас.
  Только **чистое функциональное описание, без технических деталей** (см. §4).
- **Папка `changes/`** рядом с главным файлом — по одному файлу на каждое
  изменение функционала. Имя файла: `YYYY-MM-DD-короткое-название.md`
  (дата + короткое имя добавляемой фичи).
- **Папка `general/`** — для change-запросов, которые **не относятся к
  конкретному разделу/домену** (сквозные/общие изменения), под которые нельзя
  завести отдельную папку раздела. У `general/` **нет главного `<Раздел>.md`** —
  только `changes/`. Change-файл кладётся в
  `docs/general/changes/YYYY-MM-DD-<feature>.md`.

## 2) Порядок работы с функционалом

1. **Определи, куда положить спецификацию.** Glob по `docs/**/*.md`, определи
   папку раздела, к которому относится функционал.
   - Подходящий раздел найден → используй его.
   - Раздела нет, но функционал — **новый крупный раздел** → это новый раздел
     (см. §5).
   - Функционал **не относится к конкретному разделу/домену** (сквозное/общее
     изменение, под которое нельзя завести папку раздела) → используй
     `docs/general` (см. §1); change-файл кладётся в `docs/general/changes/`.
   - **Не можешь однозначно определить раздел** (функционал подходит сразу под
     несколько разделов, либо непонятно — это `general` или конкретный раздел) →
     **спроси пользователя, куда положить спецификацию, и предложи варианты**
     (подходящие существующие разделы, `general`, либо новый раздел). Дождись
     ответа, прежде чем создавать change-файл.
2. **Опиши изменение.** Создай change-файл
   `docs/<Раздел>/changes/YYYY-MM-DD-<feature>.md` (или
   `docs/general/changes/YYYY-MM-DD-<feature>.md`) по структуре из §3.
3. **Согласование (обязательный шаг).** Покажи change-спецификацию и **спроси
   пользователя, согласен ли он** с ней. Дождись явного подтверждения.
   **Не переходи к реализации без согласия.**
4. **После согласия:**
   1. **Обнови главный `<Раздел>.md`** — чистое функциональное описание (§4):
      - новый функционал → добавь новый раздел (§) в главный файл;
      - изменение существующего поведения → отредактируй соответствующие
        существующие разделы.
      - В обоих случаях **проставь в главном файле ссылку на change-файл**.
      - **Для `general` главного файла нет** — этот подшаг пропускается, change
        живёт только в `docs/general/changes/`.
   2. **Затронуты ли другие разделы?** Если изменение из change-спецификации
      меняет поведение и других разделов, **обнови главные `<Раздел>.md` всех
      затронутых разделов** вместе с текущим (по правилам §4) и проставь в
      каждом ссылку на тот же change-файл. Главные ТЗ всех затронутых разделов
      должны остаться взаимно согласованными.
      - Затронутого раздела может **ещё не существовать** — тогда к нему
        применяй §5 (реши, создавать ли новый раздел и главный файл, или
        дописать в существующий).
   3. **Реализуй функционал** в коде. По ходу реализации **отмечай задачи из
      раздела «Код» change-файла выполненными** (`[ ]` → `[x]`): change-файл
      хранит актуальное состояние реализации, чтобы при возврате к задаче было
      видно, что уже сделано, а что осталось.

## 3) Структура change-файла

Change-файл описывает конкретное изменение и **может содержать технические
детали**. Включает:

- **Описание изменения** — что меняется и зачем.
- **Структура данных** — таблицы, типы, миграции (если есть изменения).
- **Интерфейс** — поля формы, таблицы, фильтры.
- **Бизнес-логика и валидация.**
- **Права доступа (RBAC)** — если есть изменения.
- **Код (FSD)** — таблица создаваемых/изменяемых файлов. **Первым столбцом —
  чекбокс готовности** (`[ ]`/`[x]`), чтобы по ходу реализации отмечать
  выполненные файлы и хранить состояние (см. §2).

## 4) Главный файл раздела — правила

- **Только функциональное, чистое описание**: как раздел работает с точки зрения
  пользователя и бизнес-правил.
- **Никаких технических деталей**: имён файлов, типов, SQL, эндпоинтов, имён
  компонентов, названий стека.
- Описывает **актуальное состояние**, а не историю. История изменений — в
  change-файлах, на которые главный файл ссылается.

## 5) Когда создавать новый раздел

Новая папка раздела и главный файл создаются, только если функционал — **новый
крупный раздел** системы (новая страница/модуль), не относящийся ни к одному
существующему разделу. Иначе редактируй существующий главный файл.

## 6) Исключения (можно без change-файла)

- Исправление багов.
- Мелкие UI-правки (отступы, цвета).
- Рефакторинг без изменения поведения.
